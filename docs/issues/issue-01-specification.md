# issue-01: 仕様策定

## 背景

rondo はスマホで遊ぶことを前提としたブラウザゲームプラットフォームである。ひとつの基盤の上に、ソロゲームとリアルタイムゲームをプラグインとして差し込んで遊べる構造を目指す。

実装に入る前に、設計の土台となるドキュメント群を揃える。ここで設計判断を文章として固定しておくことで、以降の実装 issue が「なぜそうするか」を毎回問い直さずに進められる状態を作る。

このサービスは「ちょっと集まって、少し遊んで、また明日」という軽い時間を支えることを核に置く。勝ち負けの強度より、気軽に立ち寄れる場であることを優先する。

## スコープ

docs を README + design-spec + directory-structure + glossary + adr/ の三層構成として確立する。

### このissueでやること

- design-spec.md を作成する（全体設計の仕様書）
- directory-structure.md を作成する（ディレクトリ構成の意図の説明）
- glossary.md を作成する（用語集）
- adr/ に設計判断を個別ファイルとして起こす（0001〜0012）

### このissueでやらないこと

- コードの実装（issue-03 以降で対応）
- 開発基盤の構築（issue-02 で対応）
- bootstrap.toml の作成（issue-02 で対応）

## 設計方針

### 判断は ADR に分けて刻む

- 設計上の決定はひとつずつ独立した ADR として記録する
- 各 ADR は「文脈 / 決定 / 理由 / 検討した代替案 / 結果」の構成で書く
- 受理した ADR は書き換えない。覆る場合は新しい ADR で置換する

### design-spec は全体像、ADR は個別判断

- design-spec は「何をどう作るか」の全体像を語る
- 個々の「なぜその判断をしたか」は ADR に委ね、design-spec からは参照する
- 二重管理を避け、判断の出どころを ADR に一本化する

### 用語を先に固定する

- ソロゲーム / リアルタイムゲーム / マニフェスト / registry / ルームアクター / 接続アクター / サーバー権威 / 契約（contracts）などの語を glossary で定義する
- 以降のドキュメントとコードは glossary の語に従う

## 受け入れ条件

- docs/design-spec.md が章立て（骨子の第2章に対応する全12章）に沿って書かれている
- docs/directory-structure.md がディレクトリ構成とその意図を説明している
- docs/glossary.md が主要な用語を定義している
- docs/adr/ に 0001〜0012 の ADR が個別ファイルとして存在する
- 各 ADR が文脈・決定・理由・代替案・結果の構成を満たしている
- design-spec から関連 ADR への参照が貼られている

## 段階

2. ADR 0001〜0012 を個別ファイルとして起こす
3. directory-structure.md を書く
4. design-spec.md を章立てに沿って書き、ADR を参照する
5. README とドキュメント間の用語・記述の整合を確認する
