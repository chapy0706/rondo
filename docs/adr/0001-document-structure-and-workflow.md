# ADR 0001: ドキュメント構成と開発フローを定める

- ステータス: 受理
- 日付: 2026-06-24

## 文脈

rondo はチームで開発し、実装は Claude Code を中心に進める。設計判断が会話やコードに散らばると、数ヶ月後に「なぜそう決めたか」をたどれなくなる。判断の出どころを一本化し、ドキュメントと実装が同じ語彙で進む状態を最初に決めておきたい。

## 決定

- docs は README + design-spec + directory-structure + glossary + adr/ の三層構成とする
- 開発は issue 駆動で進める。issue-01 を仕様策定、issue-02 を Claude Code 用開発基盤に固定し、03 以降をアプリ開発とする
- 設計判断はひとつずつ独立した ADR として記録する。ADR は受理後に書き換えない。覆る場合は新しい ADR を起こして置換する

## 理由

- design-spec に全体像、ADR に個別判断と分けることで、判断の経緯が本文に埋もれない
- ADR を不変とすることで、過去の判断が後から静かに改変されることを防ぎ、決定の履歴がそのまま残る
- issue 番号を固定することで、リポジトリを横断しても「最初の2本は仕様と基盤」という共通理解が保てる

## 検討した代替案

- design-spec に判断理由まで全部書く案。本文が肥大化し、決定の経緯と現在の仕様が混ざって読みにくくなるため却下した

## 結果

- すべてのドキュメントとコードは glossary の語彙に従う
- design-spec は関連する ADR を参照する形で書く
