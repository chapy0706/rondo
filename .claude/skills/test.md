# スキル: テスト

## いつ使うか

判定ロジック・状態遷移・契約を実装するとき。これらはテストファーストで書く。

## 原則

- ゲームルール、ルーム状態遷移、勝敗判定はテストファースト（TDD）
- フロントは Vitest、Gleam は gleam test
- 契約（contracts）と Gleam protocol の対応は、ずれを検出するテストで守る
- `make verify` が TS と Gleam の両方のテストを通すゲートになる

## 何をテストするか

- ドメインの判定: 勝敗、順位（サーバー受信順。ADR 0014）、最小人数割れ（ADR 0013）
- ルーム状態遷移: 作成 → 参加 → 終了 → 解散（ADR 0017）
- 再接続猶予: 10秒内の復帰は離脱扱いにしない（ADR 0013）
- 契約の対応: messages.ts と protocol/message.gleam の種別が一致する

## 何をテストしないか

- 描画やアニメーションの見た目
- 外部サービス（A1 / Coolify）そのものの挙動

## やってはいけないこと

- 判定ロジックを実装してからテストを後付けする。先にテストを書く
- サーバー受信順の判定を、クライアント時刻を使ったテストで通す（ADR 0014 違反）
